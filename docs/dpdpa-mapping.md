# DPDPA 2023 Obligation Mapping

This document maps each enforced section of India's **Digital Personal Data Protection Act 2023** to the system check ARC-Hawk performs and the SQL evidence query that produces the compliance gap.

The gap report is generated by `DPDPAObligationService.BuildGapReport` and is available via `GET /api/v1/compliance/dpdpa/gap-report`.

---

## Obligation Table

| Section | Obligation | System Check | Evidence Query |
|---------|-----------|-------------|----------------|
| Sec 4 | Consent before processing | Assets with PII findings must have a linked consent record (`classifications.requires_consent = false` or consent record present) | `SELECT DISTINCT a.id, a.name FROM assets a JOIN findings f ON f.asset_id = a.id JOIN classifications cl ON cl.finding_id = f.id WHERE cl.requires_consent = true GROUP BY a.id, a.name` |
| Sec 5 | Declared purpose | `declared_purpose` column (or `file_metadata->>'declared_purpose'`) must be non-null on all assets that have findings | `SELECT a.id, a.name FROM assets a WHERE EXISTS (SELECT 1 FROM findings f WHERE f.asset_id = a.id) AND COALESCE(a.declared_purpose, a.file_metadata->>'declared_purpose') IS NULL` |
| Sec 6 | Consent validity | Consent records must be present, not withdrawn, and not expired | `SELECT a.id, a.name FROM assets a JOIN consent_records c ON c.asset_id = a.id WHERE c.consent_withdrawn_at IS NOT NULL OR (c.consent_expires_at IS NOT NULL AND c.consent_expires_at < NOW())` |
| Sec 8 | Data accuracy | Assets must be re-scanned within 90 days to confirm data accuracy | `SELECT a.id, a.name, MAX(sr.scan_completed_at) AS last_scan FROM assets a LEFT JOIN findings f ON f.asset_id = a.id LEFT JOIN scan_runs sr ON sr.id = f.scan_run_id GROUP BY a.id, a.name HAVING MAX(sr.scan_completed_at) < NOW() - INTERVAL '90 days' OR MAX(sr.scan_completed_at) IS NULL` |
| Sec 9 | Children data protection | Assets with age-indicator classifications (`AGE_INDICATOR` category or sub-category containing `age`, `dob`, `date_of_birth`) must be reviewed | `SELECT DISTINCT a.id, a.name FROM assets a JOIN findings f ON f.asset_id = a.id JOIN classifications cl ON cl.finding_id = f.id WHERE cl.dpdpa_category = 'AGE_INDICATOR' OR cl.sub_category ILIKE '%age%' OR cl.sub_category ILIKE '%dob%' OR cl.sub_category ILIKE '%date_of_birth%'` |
| Sec 10 | Data fiduciary accountability | High-risk assets (risk score >= 60) must have a DPO assigned (`dpo_assigned` column or `file_metadata->>'dpo_assigned'`) | `SELECT a.id, a.name, a.risk_score FROM assets a WHERE a.risk_score >= 60 AND COALESCE(a.dpo_assigned, a.file_metadata->>'dpo_assigned') IS NULL` |
| Sec 17 | Retention enforcement | Assets with PII must have `retention_policy_days` set; findings must not exceed the retention deadline | `SELECT a.id, a.name FROM assets a WHERE a.retention_policy_days IS NULL AND EXISTS (SELECT 1 FROM findings f WHERE f.asset_id = a.id)` — plus `SELECT DISTINCT asset_id, asset_name FROM retention_violations` for deadline breaches |

## Section Detail

### Section 4 — Lawful Processing

**Obligation:** Personal data may only be processed with the data principal's consent or another specified lawful basis.

**System check:** For every asset that has at least one finding with `classifications.requires_consent = true`, the check verifies that a consent record exists. If no consent record is linked the asset is flagged as `fail`.

**Evidence query:**

```sql
SELECT DISTINCT a.id, a.name,
    BOOL_OR(cl.requires_consent) AS requires_consent,
    ARRAY_AGG(f.id ORDER BY f.id) FILTER (WHERE cl.requires_consent) AS evidence_ids
FROM assets a
JOIN findings f ON f.asset_id = a.id
JOIN classifications cl ON cl.finding_id = f.id
GROUP BY a.id, a.name
HAVING COUNT(f.id) > 0;
```

Assets where `requires_consent = true` and no consent record is linked are compliance gaps.

### Section 5 — Purpose Limitation

**Obligation:** Personal data may only be processed for the purposes specified at the time of consent collection.

**System check:** Every asset that has findings must declare its processing purpose. The check reads `assets.declared_purpose` (dedicated column added in migration 000033) and falls back to `file_metadata->>'declared_purpose'` for older deployments.

**Evidence query:**

```sql
SELECT a.id, a.name,
    COALESCE(a.declared_purpose, a.file_metadata->>'declared_purpose') AS declared_purpose
FROM assets a
WHERE EXISTS (SELECT 1 FROM findings f WHERE f.asset_id = a.id)
  AND COALESCE(a.declared_purpose, a.file_metadata->>'declared_purpose') IS NULL;
```

**Remediation:** Set the declared purpose on the asset via `PUT /api/v1/assets/{id}` with the `declared_purpose` field, or apply a tag via the bulk-tag endpoint.

---

### Section 6 — Consent Validity

**Obligation:** Consent must be freely given, specific, informed, and unambiguous. Withdrawn or expired consent is invalid.

**System check:** Delegates to the Sec 4 signal (same underlying data). A separate obligation entry is generated in the gap report for granular reporting. Additionally, the `consent_status_view` exposes a computed `status` field:

```sql
-- consent_status_view status values:
-- VALID         — consent is active and not expired
-- WITHDRAWN     — consent_withdrawn_at IS NOT NULL
-- EXPIRED       — consent_expires_at < NOW()
-- EXPIRING_SOON — consent_expires_at < NOW() + INTERVAL '30 days'
```

**Evidence query (expired or withdrawn consent):**

```sql
SELECT a.id, a.name, c.status, c.consent_expires_at, c.consent_withdrawn_at
FROM assets a
JOIN consent_status_view c ON c.asset_id = a.id
WHERE c.status IN ('WITHDRAWN', 'EXPIRED');
```

### Section 8 — Data Accuracy

**Obligation:** Data fiduciaries must ensure personal data is accurate and up to date.

**System check:** Assets not re-scanned within 90 days are flagged — stale scan data means accuracy cannot be confirmed.

**Evidence query:**

```sql
SELECT a.id, a.name, MAX(sr.scan_completed_at) AS last_scan
FROM assets a
LEFT JOIN findings f ON f.asset_id = a.id
LEFT JOIN scan_runs sr ON sr.id = f.scan_run_id
GROUP BY a.id, a.name
HAVING MAX(sr.scan_completed_at) < NOW() - INTERVAL '90 days'
    OR MAX(sr.scan_completed_at) IS NULL;
```

**Remediation:** Trigger a new scan for stale assets via `POST /api/v1/scan/trigger` targeting the relevant connection.

### Section 9 — Children's Data Protection

**Obligation:** Processing of personal data of children (under 18) requires verifiable parental consent and additional safeguards.

**System check:** ARC-Hawk flags assets containing age-indicator fields. These are identified by `classifications.dpdpa_category = 'AGE_INDICATOR'` or sub-categories matching `age`, `dob`, or `date_of_birth`. Every flagged asset is a compliance gap requiring manual review.

**Evidence query:**

```sql
SELECT DISTINCT a.id, a.name, f.id AS finding_id
FROM assets a
JOIN findings f ON f.asset_id = a.id
JOIN classifications cl ON cl.finding_id = f.id
WHERE cl.dpdpa_category = 'AGE_INDICATOR'
   OR cl.sub_category ILIKE '%age%'
   OR cl.sub_category ILIKE '%dob%'
   OR cl.sub_category ILIKE '%date_of_birth%';
```

**Note:** ARC-Hawk cannot automatically determine whether a subject is a minor — age-indicator fields are flagged for human review. All Sec 9 gaps have status `fail` until manually cleared.

### Section 10 — Data Fiduciary Accountability

**Obligation:** Significant data fiduciaries must appoint a Data Protection Officer (DPO).

**System check:** Assets with a risk score of 60 or above are considered high-risk and must have a DPO assigned. The check reads `assets.dpo_assigned` (dedicated column) and falls back to `file_metadata->>'dpo_assigned'`.

**Evidence query:**

```sql
SELECT a.id, a.name, a.risk_score,
    COALESCE(a.dpo_assigned, a.file_metadata->>'dpo_assigned') AS dpo_assigned
FROM assets a
WHERE a.risk_score >= 60
  AND COALESCE(a.dpo_assigned, a.file_metadata->>'dpo_assigned') IS NULL;
```

**Remediation:** Assign a DPO via `PUT /api/v1/assets/{id}` with `"dpo_assigned": "dpo@example.com"`.

### Section 17 — Retention Enforcement

**Obligation:** Personal data must not be retained beyond the period necessary for the specified purpose.

**System check:** Two sub-checks run:

1. **No policy defined:** Assets with findings but no `retention_policy_days` set.
2. **Policy violated:** Assets where findings are older than the configured retention window (queried from the `retention_violations` view).

**Evidence query — no policy:**

```sql
SELECT a.id, a.name
FROM assets a
WHERE a.retention_policy_days IS NULL
  AND EXISTS (SELECT 1 FROM findings f WHERE f.asset_id = a.id);
```

**Evidence query — policy violated:**

```sql
SELECT DISTINCT asset_id, asset_name
FROM retention_violations;
```

The `retention_violations` view is defined as:

```sql
SELECT
    a.id AS asset_id,
    a.name AS asset_name,
    f.id AS finding_id,
    f.created_at,
    calculate_deletion_due(f.created_at, COALESCE(a.retention_policy_days, 365)) AS deletion_due_at
FROM assets a
JOIN findings f ON f.asset_id = a.id
WHERE calculate_deletion_due(f.created_at, COALESCE(a.retention_policy_days, 365)) < NOW();
```

**Remediation:** Set `retention_policy_days` via `PUT /api/v1/assets/{id}` or trigger deletion of overdue findings via the remediation module.

## Running the Gap Report

```bash
# Full gap report (all sections)
curl http://localhost:8080/api/v1/compliance/dpdpa/gap-report | jq .

# Filter to failing obligations only
curl http://localhost:8080/api/v1/compliance/dpdpa/gap-report \
  | jq '[.data.gaps_by_section | to_entries[] | select(.value[].status == "fail")]'
```

The report is re-generated on every request — there is no caching. For large tenants (>10k assets), allow up to a few seconds for the response.

## Gap Status Reference

| Status | Meaning |
|--------|---------|
| `pass` | The obligation is satisfied for this asset |
| `fail` | The obligation is violated — action required |
| `unknown` | Insufficient data to make a determination |
