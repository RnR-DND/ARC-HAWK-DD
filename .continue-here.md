## Session 17 — Finding-Evidence Bug Fix (IN PROGRESS)

### User report
"In finding details, matching evidence is something else and field is email" — for PII detected inside a long sentence (e.g. `EMP-999888` buried in "tel, lives in Mumbai near Marine Drive. See badge EMP-999888 for details."), the UI showed the whole sentence as evidence, not the actual match.

### Root cause
`ClassifiedFinding` never carried the raw regex match — only `ValueHash` and `ContextExcerpt` (±50 chars surrounding). Backend's `SDKAdapter.MapToFinding` wrote `ContextExcerpt` into both `SampleText` (UI evidence) and the `Matches` array; `ingest_sdk_verified.go` validator also ran against `ContextExcerpt` instead of the actual match, which both leaked false positives and dropped legitimate matches whose surrounding prose failed the format validator.

### Fix (end-to-end)
1. **`apps/goScanner/internal/classifier/engine.go`** — `ClassifiedFinding` gains `MatchedValue string`. Both the built-in pattern loop and custom-pattern loop populate it with the exact regex hit `m`.
2. **`apps/goScanner/internal/orchestrator/orchestrator.go`** — `runPresidio` already had `matched := rec.Value[e.Start:e.End]`; now assigns it to `MatchedValue`.
3. **`apps/goScanner/internal/orchestrator/ingest.go`** — `buildPayload` sends `matched_value: f.MatchedValue` alongside the existing `context_excerpt`.
4. **`apps/backend/modules/scanning/service/sdk_adapter.go`** — `VerifiedFinding` gets `MatchedValue string`. New helper `vf.EvidenceValue()` returns `MatchedValue` with an `ContextExcerpt` fallback for legacy payloads. `MapToFinding` uses `evidence` for both `Matches[]` and `SampleText`; keeps `ContextExcerpt` available under `context.excerpt` for disambiguation.
5. **`apps/backend/modules/scanning/service/ingest_sdk_verified.go`** — format validator now runs on `vf.EvidenceValue()` instead of `vf.ContextExcerpt`.

### Verified
Fresh scan against pii_demo with rows containing embedded PII:
- Before: `sample_text` = "tel, lives in Mumbai near Marine Drive. See badge EMP-999888 for details." (73 chars)
- After:  `sample_text` = "EMP-999888" (10 chars); `context.excerpt` retains full 73-char context

Scanner tests pass. Backend and scanner images rebuilt and redeployed.

### Known follow-on (not from this fix)
Large scans (>6000 findings) drop ~3 ingest chunks with `Post "http://backend:8080/api/v1/scans/ingest-verified": EOF`. Backend's HTTP read timeout is too short for 2000-finding payloads. Tune backend's HTTP server `ReadTimeout`/handler context, or lower `ingestChunkSize` in `apps/goScanner/internal/orchestrator/ingest.go`. Not blocking the evidence-field fix.
