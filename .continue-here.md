**Expected:** legend lists the actual PII types present in the current graph (`EMAIL_ADDRESS`, `IN_PAN`, `IN_AADHAAR`, `IN_PHONE`, `USR_EMPLOYEE_ID`, ...) with the color each node type uses in the graph.

**Approach for next session:**
1. Pass the rendered `graphNodes` into `GraphLegend` as a prop.
2. Collect unique `pii_type` values from each node where `node.type === 'pii_category'` (look in `metadata.pii_type`).

### 2. Asset shows 7 PAN findings but lineage shows 1
**Symptom:** Asset detail page (presumably `/assets/[id]`) reports 7 findings for pattern `PAN`. Navigating to lineage shows only 1 PAN finding for that asset.

**Don't know the root cause yet.** Hypotheses to check in order:
1. **Dedup by `value_hash`** — backend's IngestSDKVerified or a downstream aggregator may collapse identical PAN values across rows. If the asset has 7 rows with the same PAN, findings table keeps 7 but lineage's pii_category node has `finding_count: 1`. Check `apps/backend/modules/scanning/service/ingest_sdk_verified.go` and the lineage aggregation in `apps/backend/modules/lineage/service/semantic_lineage_service.go`.
2. **Lineage counts only unique `pii_category` nodes, not underlying finding rows** — the lineage graph aggregates at the category level; `piiNode.metadata.finding_count` may be a distinct-count not a row-count.
3. **Frontend `piiGroups.findingCount` is derived from `piiNode.metadata.finding_count`** in `page.tsx:206`. If backend emits `count: 1` for the PII node, the frontend shows 1 — but the `fields` array populated from findingsApi.getFindings could still list 7 rows. Check whether the expand-card shows 7 rows while the summary number shows 1.

**Start next session by running:**
```sql
SELECT pattern_name, COUNT(*) FROM findings WHERE pattern_name='PAN' GROUP BY 1;
SELECT asset_id, pattern_name, COUNT(*) FROM findings WHERE pattern_name='PAN' GROUP BY 1,2;
```
Then check the lineage API response for that asset:
```
curl -sS "http://localhost:8080/api/v1/lineage?assetId=<ID>" | jq '.hierarchy.nodes[] | select(.type=="pii_category")'
```
Compare `finding_count` in the lineage node vs raw count in findings table. Fix at whichever layer lies.
