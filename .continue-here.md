## Session 10 ‚Äî Change 3: Parallel Multi-DB Scanning (DONE)

### What Was Implemented
1. **Parallel execution** ‚Äî `ThreadPoolExecutor` runs hawk_scanner across multiple source types concurrently (up to 4 workers) when `execution_mode: "parallel"`.
2. **Per-source PII config** ‚Äî Different PII types per source in a single scan. When `pii_types_per_source` is provided, each profile gets its own connection YAML with profile-specific PII types. Sources not in the per-source map fall back to global `pii_types`.

### Files Changed (uncommitted)
- **`apps/backend/modules/scanning/service/scan_service.go`** ‚Äî Added `PIITypesPerSource map[string][]string` field to `TriggerScanRequest`
- **`apps/backend/modules/scanning/api/scan_trigger_handler.go`** ‚Äî Passes `pii_types_per_source` to scanner API payload when present
- **`apps/scanner/scanner_api.py`** ‚Äî Per-profile YAML generation when `pii_types_per_source` is set; scan units abstraction (per-profile or per-source-type); temp file cleanup for all per-profile configs
- **`apps/frontend/components/scans/ScanConfigModal.tsx`** ‚Äî Toggle for per-source PII config; per-source PII type selectors with All/None; validates each source has PII types before allowing scan

### Data Flow (Per-Source PII)
```
Frontend { pii_types_per_source: { "my-pg": ["PAN"], "my-mysql": ["EMAIL","PHONE"] } }
  ‚Üí Backend scan_trigger_handler.go ‚Üí scanner_api.py
    ‚Üí writes /tmp/connection_{scan_id}_{profile_name}.yml per profile
      ‚Üí each YAML has profile-specific pii_types
        ‚Üí hawk_scanner runs per-profile with its own config
          ‚Üí findings merged into single output
```

## Session 10 ‚Äî Change 2: Vault Integration (DONE)

### What Was Implemented
HashiCorp Vault KV v2 integration for credential lifecycle management. Credentials are now dual-written to Vault (when enabled) and PostgreSQL (always, as fallback). At scan time, credentials are read from Vault first, falling back to PostgreSQL AES decryption.

### Files Changed (uncommitted)
- **`apps/backend/modules/shared/infrastructure/vault/vault_client.go`** (NEW) ‚Äî Vault KV v2 client with Read/Write/Delete/HealthCheck. Returns `ErrVaultDisabled` when not enabled so callers fall back gracefully.
- **`apps/backend/modules/shared/interfaces/module.go`** ‚Äî Added `VaultClient *vault.Client` to `ModuleDependencies`
- **`apps/backend/cmd/server/main.go`** ‚Äî Initializes Vault client between Phase 3 and Phase 4, injects into `baseDeps`
- **`apps/backend/modules/connections/service/connection_service.go`** ‚Äî `AddConnection` dual-writes to Vault + PG. `GetConnectionWithConfig` reads Vault first. `DeleteConnection` cleans up Vault secret.
- **`apps/backend/modules/connections/module.go`** ‚Äî Passes `deps.VaultClient` to `NewConnectionService`
- **`apps/backend/modules/scanning/api/scan_trigger_handler.go`** ‚Äî `resolveConnectionConfigs` tries Vault before PostgreSQL decryption
- **`apps/backend/modules/scanning/module.go`** ‚Äî Passes `deps.VaultClient` to `NewScanTriggerHandler`
- **`docker-compose.yml`** ‚Äî Added `vault` service (hashicorp/vault:1.15 dev mode), added Vault env vars to backend
- **`apps/backend/.env.example`** ‚Äî Added Vault configuration section
- **`apps/backend/go.mod`** / **`go.sum`** ‚Äî Added `github.com/hashicorp/vault/api v1.23.0`
- **`helm/arc-hawk-dd/values.yaml`** ‚Äî Added Vault section (disabled by default)
- **`helm/arc-hawk-dd/values.prod.yaml`** ‚Äî Added Vault prod overrides (external cluster)

### Data Flow
```
VAULT_ENABLED=false (default):
  AddConnection ‚Üí AES encrypt ‚Üí PostgreSQL (unchanged behavior)
  Scan trigger ‚Üí PostgreSQL ‚Üí AES decrypt ‚Üí scanner API

VAULT_ENABLED=true (Vault-only ‚Äî no credentials in PostgreSQL):
  AddConnection ‚Üí Vault KV v2 write (PG stores metadata only, no encrypted config)
  Scan trigger ‚Üí Vault read ‚Üí scanner API (no PG fallback)
  DeleteConnection ‚Üí Vault delete + PostgreSQL metadata delete
```

### Vault Secret Path Convention
`secret/data/connections/{source_type}/{profile_name}`

### How to Enable
```bash
# In .env:
VAULT_ENABLED=true
VAULT_ADDR=http://localhost:8200
VAULT_TOKEN=arc-hawk-dev-token

# docker-compose up vault  (starts dev-mode Vault)
```

## Session 9 ‚Äî Change 1: Selective PII Scanning (DONE)

### What Was Implemented
Selective PII scanning ‚Äî user-selected PII types now filter both regex patterns AND Presidio NLP entities BEFORE scanning, instead of running all 90+ patterns on every cell and filtering after.

### Files Changed (uncommitted)
- **`apps/scanner/sdk/pii_type_mapping.py`** (NEW) ‚Äî Single source of truth mapping frontend PII names ‚Üí fingerprint.yml pattern names + Presidio entity types. Includes `UPI_ID` alias for frontend compatibility.
- **`apps/scanner/hawk_scanner/internals/system.py`** ‚Äî `match_strings()` now:
  - Accepts optional `pii_types` parameter
  - Auto-reads `pii_types` from connection YAML when not explicitly passed (covers ALL 30+ connectors automatically)
  - Filters fingerprint patterns via `filter_fingerprint_patterns()`
  - Passes specific `entities=` list to Presidio instead of `None`
- **`apps/scanner/hawk_scanner/commands/postgresql.py`** ‚Äî Reads `pii_types` from connections, passes explicitly to `match_strings()` (optimization: avoids re-reading YAML per cell)
- **`apps/scanner/hawk_scanner/commands/mysql.py`** ‚Äî Same as postgresql
- **`apps/scanner/scanner_api.py`** ‚Äî Passes `config['pii_types']` into hawk_scanner YAML config
- **`apps/frontend/components/ScanAllButton.tsx`** ‚Äî Fixed to send short PII names (`AADHAAR`) instead of Presidio entity names (`IN_AADHAAR`), consistent with ScanConfigModal

### Data Flow
```
Frontend ["PAN","AADHAAR"] ‚Üí Backend scan_service.go ‚Üí scanner_api.py
  ‚Üí writes pii_types to /tmp/connection_{scan_id}.yml
    ‚Üí hawk_scanner reads YAML ‚Üí match_strings() auto-reads pii_types
      ‚Üí Fingerprint: runs only 2 regex patterns (not 90+)
      ‚Üí Presidio: entities=["IN_PAN","IN_AADHAAR"] (not None/all)
        ‚Üí ingest_results post-filter (safety net, already existed)
```

### How All 30+ Connectors Get Filtering
`match_strings()` auto-reads `pii_types` from the connection YAML (which it already parses for the `redacted` flag). No per-connector code changes needed. PostgreSQL/MySQL have explicit pass-through as a minor optimization.
