## Session 15 ‚Äî Go Scanner Wire-Up (DONE)

### What was wrong before
1. `apps/scanner/scanner_api.py` (Python scanner) was deleted during migration; its running container (`arc-platform-scanner`) kept serving a stale binary with a phantom-inode `scanner_api.py` (bind-mount deletion).
2. `apps/goScanner/` had no `Dockerfile` ‚Äî the new service in `docker-compose.yml` couldn't build.
3. Backend dispatched scans to `http://go-scanner:8001` which didn't resolve.
4. Go scanner's `ScanRequest` didn't accept the real backend payload (`sources` as `[]string` profile names, `connection_configs`, `pii_types_per_source`, `execution_mode`, `tenant_id`, etc.).
5. Orchestrator had no concept of sequential mode or per-source PII filtering.
6. Classifier didn't filter patterns by the caller's `pii_types` allowlist.
7. Ingest endpoint `/api/v1/scans/ingest-verified` required `X-Tenant-ID` which scanner never sent.
8. Backend's `TestConnectionService` decrypted only from PostgreSQL ‚Äî ignored Vault (Session 14 bug).

### What was fixed
- **New `apps/goScanner/Dockerfile`** (multi-stage Alpine runtime, `wget`-based healthcheck, static binary).
- **`apps/goScanner/api/handler.go`** rewritten to accept the backend payload shape verbatim. `sources` is polymorphic (`[]string` OR `[]SourceConfig`) to stay test-compatible. Resolves profile names via `connection_configs`.
- **`apps/goScanner/internal/orchestrator/orchestrator.go`** adds `ExecutionMode` (parallel/sequential), `GlobalPIITypes`, `PIITypesPerSource`; populates `DataSource/Host/Table` on every finding after Classify.
- **`apps/goScanner/internal/classifier/engine.go`** `Classify` now takes an `allowedPatterns` set (built from frontend PII names via `pii_types.go`).
- **`apps/goScanner/internal/classifier/pii_types.go`** maps frontend names (`PAN`, `EMAIL`, `AADHAAR`, ‚Ä¶) ‚Üí internal `Pattern.Name` values + Presidio entity labels (Presidio transport not yet wired).
- **`apps/goScanner/internal/orchestrator/ingest.go`** rewritten: chunk 2000, 2s pause, `X-Tenant-ID` header, envelope shape matches backend `VerifiedScanInput` (nested `source`, `ml_confidence` as float, correct field names).
- **`apps/goScanner/internal/connectors/databases/config.go`** new `cfgString(...)` helper; all DB connectors (postgres/mysql/mssql/mongodb/couchdb/redis) now accept both `dbname` and `database` keys, both `user` and `username`, and coerce JSON numeric ports.
- **Backend**: `scan_trigger_handler.go` includes `tenant_id` in the scanner payload. `auth_middleware.go` dev-mode branch honors `X-Tenant-ID` header (falls back to `DevSystemTenantID`, not `uuid.Nil`, so `EnsureTenantID` stops rejecting the request).
- **`apps/backend/modules/connections/service/connection_test_service.go`** now reads from Vault first, PG decrypt as fallback (Session 14 regression fix).
- **`docker-compose.yml`** go-scanner moved from IP `.40` to `.41` (the orphan `arc-platform-scanner` still holds `.40`; removing the orphan requires explicit user permission). `BACKEND_URL=http://backend:8080` added so scanner can post findings back.
- **`.env`** adds Vault block with `VAULT_ENABLED=true`.

### Verified end-to-end
Scan run `3fc521b5‚Ä¶` (parallel, all PII types): **8 findings persisted** (3 Email + 4 Phone India + 1 PAN) ‚Äî the drop from 12 is backend-side validator/dedup behavior, not scanner.

Scan run `62577d78‚Ä¶` (sequential, `pii_types_per_source={"selfscan-pg":["EMAIL"]}`): **6 Email findings only** ‚Äî per-source PII filter correctly restricts pattern set.

### Known open items
1. Orphan `arc-platform-scanner` container still running on IP `.40` ‚Äî needs user approval to `docker stop && docker rm`.
2. `go-scanner` has no Presidio integration. Python scanner called `http://presidio-analyzer:3000` for NER. Regex engine covers the primary Indian PII types (PAN, Aadhaar, Phone, Email, UPI, IFSC, GST, Passport) so Presidio is not strictly required, but contextual detection (names, addresses, DOB) is weaker without it.
3. Non-DB connectors (s3, gcs, kafka, slack, ‚Ä¶) haven't been exercised against real targets in this session; config-key compatibility fixes only applied to DB connectors. Watch for the same `dbname`/`database` type of mismatch in other connectors when someone scans a cloud source.
